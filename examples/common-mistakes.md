# 常見錯誤 + 解法

> 這些是使用 agentic 工具時最常踩的坑，附上具體的解法。

---

## 錯誤 1：CLAUDE.md 越寫越長，捨不得刪

**症狀**：CLAUDE.md 超過 200 行，裡面什麼都有，從寫程式規則到個人偏好到歷史教訓。

**為什麼會這樣**：每次 agent 犯錯，你就加一條規則。加著加著就膨脹了。

**解法**：
1. 把 CLAUDE.md 裡「不是每次都需要」的內容搬到獨立檔案
2. CLAUDE.md 只留路由邏輯：「做 X 時讀 Y」
3. 定期清理：讓 agent 幫你找冗餘和矛盾

---

## 錯誤 2：裝了一堆 plugin 但不清楚它們注入了什麼

**症狀**：agent 表現不穩定，有時很聰明有時很蠢。你不確定是 agent 的問題還是某個 plugin 的問題。

**為什麼會這樣**：每個 plugin 都會注入自己的 system prompt。你看不到，但它們在消耗 agent 的 context。

**解法**：
1. 把所有 plugin 停用
2. 用最基本的 CLI 做一個你熟悉的任務
3. 一個一個把 plugin 加回來，每加一個就測一次
4. 如果某個 plugin 加回來後表現明顯變差 → 不要用它

---

## 錯誤 3：把 rule 和 skill 搞混

**症狀**：你的 rule 檔案裡有完整的步驟指南（step 1, step 2, step 3...），或者你的 skill 裡放了一堆「不要做 X」。

**區分方法**：
- **Rule = 偏好**：「我不喜歡 agent 做什麼」或「做某件事時要注意什麼」
- **Skill = 配方**：「完成某件事的具體步驟」

**解法**：
- Rule 裡的完整步驟 → 搬到 skill
- Skill 裡的偏好設定 → 搬到 rule

---

## 錯誤 4：讓 agent 自己決定要不要讀規則

**症狀**：你在 CLAUDE.md 寫了「如果你覺得相關就讀 X」，但 agent 經常跳過。

**為什麼會這樣**：「如果你覺得相關」是一個模糊的條件。Agent 為了效率（或 sycophancy）可能會判斷「不相關」然後跳過。

**解法**：用明確的觸發條件：
```markdown
# 不好的寫法
如果你覺得任務跟寫程式有關，可以讀 coding-rules.md

# 好的寫法
寫程式（包括建立、修改、debug 任何程式碼檔案）→ 讀 rules/coding-rules.md
```

---

## 錯誤 5：Compaction 後 agent 行為失常

**症狀**：長對話中途，agent 突然開始做一些你之前已經糾正過的錯誤。

**為什麼會這樣**：Claude 在壓縮長對話時，可能會丟失你在對話中給的臨時指令。

**解法**：
1. 重要的指令不要只在對話中說，要寫進 rule 檔案
2. 在 CLAUDE.md 的啟動流程中加上 compaction 恢復步驟
3. 長任務建立任務計劃檔案，compaction 後可以重新讀取
4. 最好的解法：把大任務拆成多個小 session

---

## 錯誤 6：寫規則時用模糊的語言

**症狀**：你寫了「盡量不要覆蓋檔案」，但 agent 還是覆蓋了，然後說「我認為這次覆蓋是合理的」。

**為什麼會這樣**：「盡量」給了 agent 一個退路。它可以自行判斷什麼時候「不盡量」。

**解法**：規則要寫得像法律，不留模糊空間：
```markdown
# 不好的寫法
盡量不要覆蓋現有檔案

# 好的寫法
目標位置有同名檔案時，絕對不能覆蓋。必須：
1. Read 兩邊內容
2. 比較差異
3. 向使用者確認操作方式
4. 合併寫入
直接覆蓋是禁止行為。
```

## 錯誤 7：只看 agent 有沒有「完成」，不看它怎麼完成的

**症狀**：agent 說「完成了」，你就接受了。後來才發現它用了 stub、mock、或根本沒有真正實作。

**為什麼會這樣**：agent 知道怎麼開始任務，但不一定知道怎麼「真正」結束。它可能會在一個「看起來完成」的狀態下停止。

**解法**：
1. 用 task contract 定義明確的完成條件
2. 完成條件包含可驗證的測試
3. 告訴 agent：「除非這些條件全部滿足，否則任務不算完成」
4. 加入截圖驗證：讓 agent 截圖並確認行為符合預期
